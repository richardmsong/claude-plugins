## Findings

### Finding 1 (Minor, non-blocking): Definition truncation for `no_test_scaffolding_types` in registry.yaml

**Location**: `src/sdd/spec/registry.yaml` line 529 vs. ADR Invariant Delta definition at line 156 of the ADR.

**Observation**: The ADR's Invariant Delta carries a three-sentence definition for `methodology.validator.no_test_scaffolding_types`:

> "...returns no error when no test file defines a concrete methodology-interface-satisfying type. Detects `noopValidator`, `stubValidator`, `fakeValidator`, and any naming variant by structural method-set matching. Generic test mocks of unrelated interfaces (e.g. external libraries, HTTP clients) remain allowed because those interfaces aren't declared in `*_interface.go` files."

The registry.yaml entry stops after the first sentence ("...returns no error when no test file defines a concrete methodology-interface-satisfying type."), omitting the second and third sentences.

**Impact**: The registry definition is a valid single-line string and does not violate `methodology.validator.registry_definition_field` (non-empty, no newlines). The operational contract for the validator is unambiguously conveyed in the first sentence; the omitted sentences are informational elaboration. However, the ADR's Invariant Delta is the normative source from which the registry is populated; a divergence creates a traceability gap.

**Recommendation**: On the next registry.yaml edit for this entry, extend the definition to match the ADR's full text. Alternatively, trim the ADR's Invariant Delta to match the registry if the additional sentences are editorial rather than contractually load-bearing.

---

### Finding 2 (Cosmetic, non-blocking): Receiver type in Decisions table is value receiver; implementation uses pointer receiver

**Location**: ADR-0082, Decisions table, "Validator pattern" row.

**Observation**: The row states the method signature as `func (v validator) CheckXxx(...)` (value receiver). All existing method bodies in `src/sdd/spec/checks.go` use `func (v *validator) CheckXxx(...)` (pointer receiver), and `checks_registry.go` instantiates `v := &validator{}` (pointer).

**Impact**: Purely cosmetic. Go's interface satisfaction works correctly with pointer receivers when the concrete value is a pointer — the interface contract is unaffected. Dev-harness will implement the three new methods with pointer receivers (consistent with all 28 existing methods), which is correct behavior. The ADR's example signature is illustrative, not normative.

**Recommendation**: Correct the receiver type in the Decisions table from `(v validator)` to `(v *validator)` during the next ADR revision to prevent confusion for future implementers reading the document.
