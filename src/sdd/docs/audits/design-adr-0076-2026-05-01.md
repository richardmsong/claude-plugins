### Round 8

**Gaps found: 6 blocking**

1. **Null-guard contradicts Round 7 fix** — Round 7 said "functions accept `string | null` and no-op internally" but the ADR's Component Changes §docs-dashboard still shows `if (gitRoot) { backfillImpact(...) }` with functions typed `repoRoot: string`. The two positions are mutually exclusive. The ADR must pick one: either (a) functions are internally null-safe and `repoRoot` is `string | null`, matching `runLineageScan`/`runBlameScan` which also receive nullable `gitRoot` directly in boot.ts without a guard, or (b) keep the `if (gitRoot)` guard and type `repoRoot: string`. Currently the ADR text says both.

2. **`indexImpact` full-scan behavior unspecified** — `indexImpactFile` is specified to delete-all-for-adr + re-insert. `indexImpact` (full scan) invokes this per file, but what happens to stale rows for ADRs whose `.impact.json` was deleted between runs? The ADR never says `indexImpact` truncates the whole `adr_spec_impact` table before re-scanning. If a `.impact.json` is removed (e.g., for an ADR that had no actual spec changes), its DB rows persist forever. The full-scan behavior must be: truncate `adr_spec_impact` then re-insert from all found files, or explicitly scan for missing files and delete their rows.

3. **ImpactView props interface not defined** — Every other route component (`AdrDetail`, `AuditDetail`) has an explicit prop interface in the ADR or is derivable from the existing pattern. For `ImpactView`, the ADR says "ImpactView derives the ADR path as `${slug}.md`" and "re-fetches on any `reindex` event" — implying props are `{ slug, navigate, lastEvent }`. But `App.tsx` must extract the slug from `#/impact/<slug>` and pass it. The App.tsx route match `route.startsWith("/impact/")` + `slug = route.slice("/impact/".length)` is not stated. Without this, the implementor must infer from the AdrDetail pattern but the gap is real.

4. **"Spec impact" button conditional rendering not specified** — When `getImpact` returns `[]` for an ADR (no `.impact.json` exists or file is empty), should the button appear on AdrDetail? If the button links unconditionally to `#/impact/<slug>`, the ImpactView will show an empty right pane. This is UX-ambiguous: the ADR should state whether to always show the button or conditionally render it only when impact data exists. The ImpactView empty state is also unspecified.

5. **`backfillImpact` write failure handling** — The ADR says "Non-fatal on error" only in the spec-dashboard.md spec update section (step 7 description), not in the `src/impact.ts` Component Changes description of `backfillImpact`. The function description says it "writes `.impact.json` to disk" but specifies no error handling for I/O failures (permissions, disk full, etc.). The watcher and `indexImpact` would then see a missing file for that ADR and produce no rows, which is silently wrong. The function must catch per-ADR write errors, log a warning, and continue — same pattern as `runBlameScan`.

6. **`spec-docs-mcp.md` update content not specified** — The Impact table entry for `spec-docs-mcp.md` says "(schema version bump to '5', new `adr_spec_impact` table, watcher `.impact.json` handling, new `"./impact"` export in package exports)". Unlike the `spec-dashboard.md` Component Changes section (which has explicit table row text), there is no corresponding `spec-docs-mcp.md` Component Changes section. An implementor writing the spec update must infer the exact changes to make. The spec currently states `Current version: "4"` under Data store, and the Tables section, Exports section, and Watcher section all need specific new text — none of it is written in the ADR.
