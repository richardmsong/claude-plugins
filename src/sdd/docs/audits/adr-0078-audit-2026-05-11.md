**No composite invariants.** `methodology.registry.no_and_in_definition` catches the most common symptom (conjunction in a single definition). This is a minor gap: a consumer could register a verifier at an arbitrary path outside `spec/` and the structural checks would pass. The gap is acknowledged by design (the convention is documented; enforcement is indirect via path registration).

## 3. Invariant → Decision Roundtrip (Reverse Direction)

For each of the 31 delta entries, does it ladder up to a named Decisions table row?

30 of 31 do. One does not:

### DRIFT ITEM D-1: `methodology.adr.requires_decision_history` has no Decisions table row

**Invariant definition**: Every `adr-*.md` file under `<spec.adr_dir>` contains a `## Decision history (rationale notes)` section.

**What the Decisions table contains**: There is no row for "mandatory Decision history section" or "mandatory Decision history written-as-decided" in the 29-row table.

**Where the rationale lives**: The Decision history section itself has a paragraph starting "Why mandatory Decision history written-as-decided" that explains the rule at length. This rationale note is in the right place for its content, but the corresponding row in the Decisions table is absent.

**Impact**: The invariant operationalizes a commitment for which no Decisions table row exists. A reader using the Decisions table as the contract surface cannot reconstruct why `methodology.adr.requires_decision_history` exists — the forward direction is missing. The audit chain (Layer 1) would flag this: reconstructing decisions implied by the 31-entry constellation produces a "mandatory Decision history section per ADR" commitment that does not appear in the table.

**Suggested fix**: Add a row to the Decisions table, e.g.:

| Decision | Choice | Rationale |
|---|---|---|
| Mandatory Decision history section | Every `adr-*.md` must contain a `## Decision history (rationale notes)` section | Rationale captured retroactively is reconstructed memory; writing decisions as they are made makes context-compaction recoverable and signals whether any surprising choices arose. |

### 4c. Security section — dangling invariant citation

**Issue S-1 (advisory)**: The Security section, line 626:

> The `invariant-compiler` subagent runs in a fresh context per invocation (per `methodology.subagent.fresh_context`); no cross-invocation prompt injection surface.

`methodology.subagent.fresh_context` is cited as a registered invariant ID but does **not** exist in the delta or the registry. It is a forward-reference to an invariant that was never registered in this ADR.

**Classification**: This is a narrative commitment smuggled as an invariant citation without a delta entry. The claim about fresh-context isolation is real (it's how the subagent is designed), but it is not operationalized as a registered verifiable contract anywhere. Since ADR-0080 covers eval-as-verifier-mechanism for skill/agent behavioral contracts, `methodology.subagent.fresh_context` is likely intended for that ADR.

**Suggested fix**: Either (a) add `methodology.subagent.fresh_context` to this ADR's `### Added` block with a verifier, or (b) remove the parenthetical ID citation and replace it with prose ("by design; see ADR-0080 for agent-behavior invariants").

### 4e. Scope section

### 4f. Data Model section
