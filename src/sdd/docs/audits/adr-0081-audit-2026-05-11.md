## Findings

### Material (must fix before acceptance)

**M-1 — Scope "Deferred" section is stale (Scope)**

The "Deferred" bullet reads: "Mechanical enforcement of test-shape (i.e., 'the test bounds the validator with synthetic inputs, not with the embedded registry'). For now, this is discipline." This decision was made in Batch 3 of Open Questions: `methodology.validator.test_shape_unit_only` with AST-walker enforcement was registered. The Deferred bullet was not removed. It now contradicts the actual scope of v1.

Fix: Remove the first Deferred bullet. Update "In v1" to include `methodology.validator.test_shape_unit_only` and the test-shape enforcement.

Also: "In v1" still says "All 30 methodology invariants" — should be 31.

**M-2 — CheckError struct definition in Data Model is stale (Data Model)**

The Data Model section shows:
```go
type CheckError struct {
    Name    string
    Message string
    Path    string
}
```
The Decisions table row "CheckError shape" and `checks_interface.go` both specify:
```go
type CheckError struct {
    EntryID string
    Field   string
    Path    string
    Message string
}
```
The interface file is authoritative. The Data Model section and the Component Changes line (`"type CheckError struct{Name, Message, Path}"`) are stale drafts. Tests assert on `Field` (not `Name`); any dev-harness implementing the stale shape would break all test assertions.

Fix: Update the Data Model `CheckError` type to match the interface (EntryID, Field, Path, Message). Update the Component Changes description of `checks_registry.go`.

**M-3 — Decision row "Test-shape mechanical enforcement" has wrong verifier path (Decisions)**

The Decisions table states the verifier is `spec/checks_adr_test.go::TestCheckTestShapeUnitOnly`. The function is in `spec/checks_validator_test.go`. The Invariant Delta YAML and `registry.yaml` are both correct. The Decisions row is the only place with the wrong file.

Fix: Change `spec/checks_adr_test.go` to `spec/checks_validator_test.go` in the Decision row.

**M-4 — Error Handling commits to "distinctive error code" for empty AllChecks without a registered invariant (Error Handling)**

The Error Handling section states: "if `AllChecks` is empty..., `sdd verify` reports `structural.no_checks_registered: 0 checks defined` and exits with a distinctive error code." This is a concrete behavioral commitment (a specific output format and a specific exit code scenario) not covered by `methodology.validator.cli_exit_codes` (which only asserts non-zero exit on check failure, not on empty-AllChecks defensive path).

Fix: Either (a) register a `methodology.validator.cli_no_checks_guard` invariant, or (b) downgrade the prose to say "exits non-zero with an explanatory message (discipline, not tested)" to make clear it is not a registered contract.

Also: Implementation Plan says "31 functions: 30 ports + the AST-walker". Since there are 32 registered invariants (31 supersessions + 1 net-new), `checks.go` needs 32 methods (31 ports + 1 AST-walker). The Implementation Plan count of 31 is wrong.

**E-2 — Decision history heading and body for prefix choice are stale**

The heading "Why `methodology.validator.*` prefix **(open)** instead of keeping `methodology.<area>.*`" still carries `(open)`. The body says "Picked: open question — see Open Questions." The Open Questions section shows it was resolved in Batch 1. The heading and "Picked" line should reflect the actual decision made.

Fix: Delete the paragraph.

**E-4 — `glossary_scope_field` definition in ADR Invariant Delta YAML differs from `registry.yaml`**

The ADR delta YAML's definition says the validator flags entries whose scope is not one of `"methodology" or "consumer"` (two values). The `registry.yaml` definition and the actual test behavior (using `ScopeMethodology`, `ScopeProjectCrossCut`, `ScopeComponentLocal`) specify three valid values. The ADR delta YAML is wrong; registry.yaml is correct.

Fix: Update the ADR delta YAML definition to list all three valid values: `"methodology"`, `"project-cross-cutting"`, `"component-local"`.

**E-5 — "Test file layout" Decision row omits `checks_cross_cutting_test.go`**

The Decision row "Test file layout" lists five test files. It omits `checks_cross_cutting_test.go`, which covers 8 cross-cutting invariants and already exists on disk. The Implementation Plan (line 487) correctly includes it. The Decision row is incomplete.

Fix: Add `checks_cross_cutting_test.go` to the list in the "Test file layout" Decision row.

**E-6 — `methodology.validator.registry_id_field` requires its withdrawn predecessor (requires-graph anomaly)**

`methodology.validator.registry_id_field` has `requires: methodology.registry.id_field` (withdrawn). This edge resolves structurally (withdrawn targets are accepted by `CheckRequiresTargetsExist`) but is semantically hollow — the topo sort will include the active entry but not the withdrawn predecessor, so the requires ordering claim cannot be honored at runtime. No other new entry has this pattern. The edge appears to be a draft artifact carried from the ADR's example YAML into the registry. It does not affect runtime correctness today (the requires edge only influences dispatch order, and the withdrawn predecessor is not in AllChecks).

Fix: Remove `requires: methodology.registry.id_field` from `methodology.validator.registry_id_field` in both the ADR delta YAML and `registry.yaml`. The entry needs no requires edges.

## Self-Application Note: `cli_test.go` reads `Registry` directly

`spec/cli_test.go::TestVerifyRunsAllStructuralChecks` (line 47) reads the embedded `Registry` package-level variable directly. This is exactly the pattern `methodology.validator.test_shape_unit_only` is designed to catch. However, `cli_test.go` is in the `spec/` package, so the invariant's validator (which scans `spec/`) will flag it.

Additionally, `cmd/sdd/verify_test.go::TestRunStructuralChecksDispatchesAll` (lines 22-23) reads `spec.Registry` and `spec.Glossary`. The ADR delta YAML's definition of `test_shape_unit_only` covers `spec/ (or cmd/sdd/)`, but `registry.yaml`'s definition only mentions `spec/`. If `cmd/sdd/` is not in scope, the CLI unit test escapes the check.

The ADR acknowledges (Implementation Plan) that `spec/cli_test.go` still has `TestVerifyRunsAllStructuralChecks` and `TestVerifyWalksADRDir` which are to be removed as part of dev-harness work. Until that cleanup happens, `test_shape_unit_only` will immediately self-apply and flag `cli_test.go`. This is the correct intermediate state: the test exists to catch exactly this shape.

This note is informational rather than a finding, consistent with the foundational ADR's red-build context. Dev-harness must: (a) remove the two old functions from `cli_test.go`, and (b) resolve the definition scope gap (E-4 sibling — see below).

**Additional editorial note**: `registry.yaml`'s definition of `test_shape_unit_only` omits `cmd/sdd/` from the scan scope compared to the ADR delta YAML. If left as-is, `cmd/sdd/verify_test.go`'s `spec.Registry` reads will escape the invariant. This is a scope narrowing that should be an intentional decision (is `cmd/sdd/` in scope for the meta-check?) and, if intentional, should be documented.
